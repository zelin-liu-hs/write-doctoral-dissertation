# 博士大论文写作与优化

## 核心目标

围绕一个值得解释的核心问题，建立“问题提出—文献缺口—概念与理论—研究设计—经验识别—机制解释—结论回应”的完整证据链。把章节视为同一问题的连续求解，不把若干论文或模型简单装订。

## 必读资源

1. 每次触发后先读 `references/han-yi-doctoral-writing-method.md`，据此确定问题链、章节功能、检查清单和写作模板。
2. 需要准确使用“研究九问”、摘要五要素、目录六项要求或课堂 PPT 中的方法提醒时，读 `references/han-yi-classroom-slides-notes.md`。
3. 需要核对老师现场解释、课程边界或蒸馏稿中的不确定点时，再读 `references/han-yi-course-transcript.md`。不要把校订转写或 PPT 文字改写成未经确认的逐字引语。
4. 任何涉及引文、参考文献或事实性表述的任务，都读 `references/citation-and-evidence-rules.md`。

## 工作原则

- 默认用中文工作；保留必要的英文概念或中英对照。
- 先给结论或诊断，再给依据和修改结果。
- 区分已核查事实、文献观点、基于材料的推断和写作建议。
- 保持原文的核心概念、变量关系和因果强度；证据不足时降低表述强度。
- 不覆盖原始论文、原始数据或原始文献。所有改写稿、分析数据和输出另存。
- 不把相关性写成因果性，不用显著性替代理论解释。
- 不机械套用八章结构、混合方法、母本概念或 PSM-DID。章节和方法必须由研究问题决定。
- 把“研究九问”、摘要五要素和目录六项要求作为功能性诊断框架，不把它们误用为所有学校和学科统一的固定版式。
- 不用 CiteSpace、VOSviewer 图谱或汇总表代替原文阅读、理论比较和批判性述评。
- 不虚构文献、作者、年份、题名、期刊、页码、DOI、政策、数据、访谈、统计结果或论文内容。
- 无法核验的内容标为 `【待核验】`；需要文献但尚未找到可靠来源的句子标为 `【待补文献】`。

## 标准工作流

### 第一步：确认任务与成功标准

判断属于哪一种任务：

- 从材料起草整篇博士论文；
- 优化现有博士论文；
- 重构题目、问题链、目录或理论框架；
- 撰写或修改单章；
- 审稿式诊断研究设计、方法或证据；
- 整理引用并生成 DOCX。

明确输入材料、目标学校或学科、目标章节、篇幅、截止范围和最终文件。信息不足但可以安全推进时，先给“基于当前材料的版本”，并列出假设与缺口；可能改变研究方向、识别策略或原始材料时再询问用户。

### 第二步：盘点和分级材料

1. 列出论文正文、开题报告、研究计划、数据、访谈、政策文件、参考文献、学校格式规范和用户笔记。
2. 记录文件类型、版本、日期和用途，识别最新稿与原始稿。
3. 按以下证据层级管理材料：
   - A：已阅读全文的原始文献、官方文件、原始数据或完整访谈；
   - B：仅核验摘要、元数据或权威数据库记录；
   - C：二手综述、新闻、纪要或他人转述；
   - D：用户设想、模型推断或尚未核验的信息。
4. 读取 DOCX、PDF 或表格时保留页码、章节、表号、变量名和来源路径，避免只抽取无定位文本。

### 第三步：建立证据台账

在写作前建立最小证据表：

| 编号 | 拟写主张 | 来源 | 定位 | 核验层级 | 可支持强度 | 状态 |
|---|---|---|---|---|---|---|
| E1 | 具体主张 | 文件或文献 | 页码/章节/表号 | A/B/C/D | 描述/相关/因果 | 已核验/待核验 |

每个确定性事实、理论归属、指标定义、政策规定和因果主张都必须能回到证据台账。证据冲突时分别呈现，不强行合并。

### 第四步：重建论文问题链

先按课堂 PPT 的顺序逐项回答“研究九问”：

1. 我有一个好的问题吗？
2. 前人如何解释这个问题？
3. 我如何解释这个问题？
4. 我的解释逻辑自洽吗？
5. 我的解释是新的吗？
6. 我的解释更优吗？
7. 有经验证据支持吗？
8. 有何政策实践意义？
9. 有何理论学术贡献？

任何一问无法用一句具体、可核验的话回答，都意味着相应环节尚未完成。完成九问后，再用一页纸记录研究对象、理论或制度预期、经验事实、二者张力、一个核心问题、按认知顺序排列的子问题、三类最接近的文献及解释边界、可获得证据和有限理论增量。

建立“问题—缺口—理论—概念—证据—方法—章节—发现—结论/创新”矩阵。没有证据的问题应删除或补证；没有问题承载的章节应合并、重构或删除。

### 第五步：诊断或制定写作方案

按以下顺序审查：

1. 题目是否包含明确对象、解释切口和边界；
2. 摘要是否呈现问题、缺口、理论、数据方法、主要发现和贡献边界；
3. 绪论是否从现实变化逐步收敛到真问题；
4. 文献综述是否按解释路径、机制或争议组织；
5. 理论缺口是否属于机制、层次、情境或整合缺口；
6. 核心概念是否有定义、边界、维度、反例和识别规则；
7. 理论框架中的每个方框和箭头是否可被材料或变量观察；
8. 每章是否回答一个必要问题，并为下一章留下接口；
9. 方法是否服务于问题，识别假设是否明确；
10. 结论是否逐项回应问题，创新是否能在正文中定位。

对每个问题写明：问题是什么、为什么构成问题、如何修改、可替代方案。

摘要至少检查五个功能要素：研究背景与问题、理论视角与分析框架、研究方法、从问题到结论的推进逻辑、按问题或章节逻辑排列的主要结论。

目录至少检查六项功能：必要的学理分析章；提出—分析—解决的章节闭环；必要的对象历史、演变或现状；经验章与分析框架逐项对应；方法贯穿主体论证；结论完成问题回溯、理论对话、创新与实践启示。

### 第六步：分阶段写作

优先按最小可验证单元推进：

1. 确认问题链和目录；
2. 确认每章“五句卡片”；
3. 完成证据充分的段落；
4. 对事实和引用逐句核验；
5. 完成章节内部闭环；
6. 完成跨章节接口；
7. 最后写摘要、结论和创新点。

段落默认采用：

- 主题句：本段要回答什么；
- 证据句：文献、材料、数据或案例说明什么；
- 分析句：证据为何支持该判断；
- 边界句：结论在哪些条件下成立；
- 接口句：本段如何进入下一段或下一节。

不要用背景、口号、模型名称或作者罗列代替论证。

### 第七步：审查研究设计和因果强度

明确研究问题、因变量、自变量或政策变量、样本、时间、数据、识别策略和主要威胁。根据方法检查外生性、反向因果、遗漏变量、选择偏误、测量误差、提前反应、共同趋势、政策外溢、多重政策、处理异质性、聚类层级、机制识别和外推边界。

定性研究检查案例选择、材料饱和、负例、过程证据、编码透明度和替代解释。混合方法检查两类方法是否共享问题和构念、概念如何转化为变量、证据冲突如何处理。

方法部分不能只列名称。半结构访谈至少说明对象、样本量、核心议题和分析过程；案例或文本分析至少说明选择规则、材料来源和编码过程。把文献梳理视为研究基础和理论准备，除非目标学科规范明确认可，否则不要把“文献研究法”包装成核心经验识别方法。

多案例研究需要说明量化描述与质性阐释如何互补，并让本章发现为下一章留下需要继续回答的问题。

### 第八步：执行引用审计

1. 逐项核对文内引文与文末参考文献。
2. 给确定性陈述配置来源；常识性过渡句不滥加引用。
3. 优先使用原始全文、官方来源和权威数据库。
4. 未读全文时明确写“仅核验摘要/元数据”，不声称已阅读全文。
5. 直接引语尽量给页码，避免大段复制。
6. 学校最新规范优先；未提供学校规范时，暂按 `references/citation-and-evidence-rules.md` 的 GB/T 7714 方案输出，并在交付中标记“格式待学校规范复核”。

### 第九步：质量复核

至少完成以下检查：

- 题目、摘要、核心问题、结论使用同一组核心概念；
- 每个研究问题有证据和章节回应；
- 每个框架要素有材料、变量或明确的未检验标记；
- 数据、样本筛选、变量构造和统计结果可复核；
- 结论强度不超过证据和识别策略；
- 非显著结果、失败案例、限制和不确定性未被隐藏；
- 引用真实、可定位，参考文献与正文一一对应；
- 最终发现与初始分析框架已对照，明确哪些关系得到支持、修正或扩展；
- 原始文件未被覆盖；
- 修改稿与原稿之间有变更说明。

### 第十步：生成 DOCX

先生成结构清晰的 UTF-8 Markdown 中间稿，再运行：

```powershell
python scripts/markdown_to_docx.py <input.md> <output.docx>
```

DOCX 文件必须另存到用户指定的输出目录；若未指定，则在当前项目创建 `output/`。文件名包含论文或章节名称和版本标识。生成后用 `python-docx` 重新打开，并检查 ZIP 结构、标题层级、表格、中文字体和文件大小。

## 交付要求

除非用户只要求聊天回答，至少交付：

1. 修改后的 DOCX；
2. 可追踪的 Markdown 中间稿；
3. 简洁的修改说明；
4. 引用或证据待核验清单；
5. 已执行和未执行的验证说明。

优化现有论文时，默认保留原稿，并生成新版本，不静默替换。
